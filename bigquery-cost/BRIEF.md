**Project Brief: BigQuery FinOps Cost Analytics CLI**

## **1\. Executive Summary**

We require a local Node.js Command Line Interface (CLI) tool to audit and analyze Google BigQuery compute costs. We generate approximately $10,000/month in on-demand BigQuery spend across multiple GCP projects and regions.

The primary challenge is **Query Variance**: custom internal tooling generates thousands of queries daily that are functionally identical but structurally unique (e.g., dynamically injected UUIDs, temporary table suffixes, varying whitespace). The CLI must extract execution metadata, apply a generic programmatic normalization strategy to group these "subtly different" queries together, calculate their exact USD cost, and output actionable CSV reports.

## **2\. Technical Stack & Interface**

* **Environment:** Node.js (v18+). Run locally from the user's terminal.  
* **Libraries:**  
  * @google-cloud/bigquery (Official GCP BigQuery SDK)  
  * commander or yargs (For parsing CLI arguments)  
  * csv-writer or fast-csv (For generating CSV outputs)  
* **Authentication:** Application Default Credentials (gcloud auth application-default login).  
* **Scope:** Compute costs only (ignore storage). Reporting only (no alerting).

### **Required CLI Arguments**

The script must accept dynamic inputs to shape the scope of the FinOps audit:

* \--start-date (Required): Format YYYY-MM-DD  
* \--end-date (Required): Format YYYY-MM-DD  
* \--projects (Required): Comma-separated list of GCP Project IDs (e.g., project-a,project-b)  
* \--regions (Required): Comma-separated list of BigQuery regions (e.g., europe-west2,us)

*Example execution:*

node bq-analyzer.js \--start-date 2026-02-01 \--end-date 2026-02-28 \--projects data-prod-1,app-prod-2 \--regions europe-west2,us

## **3\. Core Logic: The Generic Normalization Engine**

Because we do not have a known list of custom-tooling string patterns, the tool must implement a generalized Regex pipeline directly in the BigQuery SQL to normalize the query text.

The SQL must generate a normalized\_blueprint column using chained REGEXP\_REPLACE functions:

1. **Truncation:** Wrap the raw query in SUBSTR(query, 1, 2000\) first to prevent the regex engine from timing out on massively long generated queries.  
2. **Strip UUIDs:** Replace r'\[0-9a-f\]{8}-\[0-9a-f\]{4}-\[0-9a-f\]{4}-\[0-9a-f\]{4}-\[0-9a-f\]{12}' with '\<UUID\>'.  
3. **Strip Dates/Timestamps:** Replace r'\\d{4}\[-\_/\]?\\d{2}\[-\_/\]?\\d{2}( \[0-9:.\]+)?' with '\<DATE\>'.  
4. **Strip Standalone Numbers (IDs):** Replace r'\\b\\d+\\b' with '\<NUM\>'.  
5. **Compress Whitespace:** Replace r'\\s+' (newlines, tabs, multiple spaces) with ' ' (a single space).

This resulting normalized\_blueprint string will be used in the GROUP BY clauses to aggregate the fragmented queries.

## **4\. Financial Math & Mandatory SQL Constraints**

To ensure accuracy and prevent the script itself from incurring massive costs, all SQL queries generated by the Node.js app must adhere to these rules:

* **Cost Formula (On-Demand):** The standard list price is $6.25 per TiB. Compute this as: (total\_bytes\_billed / POWER(1024, 4)) \* 6.25 AS estimated\_cost\_usd.

* **Mandatory Partition Pruning:** Queries against INFORMATION\_SCHEMA MUST include: WHERE creation\_time \>= TIMESTAMP('{start-date}') AND creation\_time \<= TIMESTAMP('{end-date} 23:59:59').  
* **Compute Isolation Filters:**  
  * job\_type \= 'QUERY'  
  * state \= 'DONE'  
  * cache\_hit \!= TRUE (Cached queries process 0 bytes and are free; exclude them).  
  * statement\_type \!= 'SCRIPT' (Multi-statement scripts have parent jobs that double-count child bytes; exclude the parent scripts).

## **5\. Execution Architecture (Multi-Project/Region)**

INFORMATION\_SCHEMA views are strictly regionalized and bound to specific projects. The Node.js application cannot run a single global query. It must:

1. Parse the \--projects and \--regions arrays.  
2. Iterate through every combination (Cartesian product) of Project × Region.  
3. Dynamically inject the location into the FROM clause: \`{project\_id}\`.\`region-{region}\`.INFORMATION\_SCHEMA.JOBS\_BY\_PROJECT.  
4. Execute the queries asynchronously. Wrap execution in a try/catch block—if a specific project doesn't have BigQuery enabled in a specific region, log a warning to the console but continue processing.  
5. **Aggregate the cross-project/cross-region result sets in Node.js memory** before writing out to the CSVs. (e.g., If user@domain.com spent $50 in Project A and $20 in Project B, the Node script must sum this to $70 in the final CSV).

## **6\. Required Output Reports**

The script must generate three distinct .csv files saved to a local ./reports directory:

### **Report 1: spend\_per\_user.csv**

* **Purpose:** Macro view of which human users or service accounts are driving the overall invoice.  
* **Columns:** user\_email, total\_queries\_executed, total\_tib\_billed, estimated\_cost\_usd.  
* **Sort:** estimated\_cost\_usd DESC.

### **Report 2: top\_normalized\_queries.csv**

* **Purpose:** Micro view identifying the specific systemic custom-tooling workloads draining the budget.  
* **Columns:** normalized\_blueprint, execution\_count, unique\_users, total\_tib\_billed, estimated\_cost\_usd, avg\_cost\_per\_run\_usd.  
* **Filter:** Exclude ad-hoc exploratory queries using HAVING execution\_count \> 5\.  
* **Sort:** estimated\_cost\_usd DESC. Limit to top 100\.

### **Report 3: top\_user\_query\_matrix.csv**

* **Purpose:** A matrix answering "What exactly are my high-spending users actually running?"  
* **Columns:** user\_email, query\_rank (1 through 5), normalized\_blueprint, execution\_count, estimated\_cost\_usd.  
* **Logic:** Use a SQL Window function (ROW\_NUMBER() OVER(PARTITION BY user\_email ORDER BY total\_bytes\_billed DESC)) to extract the top 5 most expensive normalized blueprints for *each* distinct user.
